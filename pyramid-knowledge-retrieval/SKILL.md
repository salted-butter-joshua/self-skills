# 金字塔知识体系检索

从知识点或一句话出发，**按金字塔逐层深入**，直到原理层。核心目标：**读者跟完一条完整链路，知道每一步为什么存在、怎么实现、如何接到下一步**；**本层所有合理疑问在本层或下层被消掉**，不留悬而未决的「然后呢？」。

## 与相近 skill

| Skill | 何时用 |
|-------|--------|
| `onion-pyramid-learn` | 单概念由外向内剥、重直觉 |
| `pyramid-knowledge-map` | 要 Mermaid 图谱、节点 YAML |
| **本 skill** | 要**疑问驱动 + 有序链路 + 为什么/如何 + 细节不丢** |

## 输入

- **知识点**或**一句话** → 提炼**可检验顶层问句**（≤30 字）
- 含糊时：≤2 种读法 → 选定及理由 → 继续

## 工作流

```
- [ ] 1. 定锚：问句、边界、原理层停止条件
- [ ] 2. 检索依据（技术类必做 WebSearch + WebFetch）
- [ ] 3. 疑问清单：列出读者会问的 10–20 个问题，映射到 L0–L4
- [ ] 4. 术语清单 → §1
- [ ] 5. 路径清单（≥2 条正交路径）→ §2
- [ ] 6. 规划有序主链（调用/时间顺序，非并列卡片）
- [ ] 7. 按 6 段版式输出；每层写「本层串讲」
- [ ] 8. §5 核对疑问清单是否全部闭合；未闭合的写入 §5 残余 FAQ
- [ ] 9. 自检
```

## 金字塔分层

| 层 | 必须回答 | 停止信号 |
|----|----------|----------|
| **L0** | 整条链路分几段、各段分工、数据/控制如何流过 | — |
| **L1** | 各段**为什么独立存在**、有哪些**正交路径/选型** | 再拆同义反复 |
| **L2** | 各段内的**关键对象**是什么、各自职责 | 可命名、可定义 |
| **L3** | **如何运作**：步骤、数据结构、接口、谁调谁 | 能按顺序复述过程 |
| **L4** | **不可再分的原理**；再拆越界 | 触达公理/架构事实 |

## 固定输出版式（6 段）

```markdown
# 金字塔检索 · {主题}

## 0｜概览与完整链路
## 1｜术语地基
## 2｜路径对比（无并列路径则从略）
## 3｜逐层深入（L0 → L1 → … → L4）
## 4｜支链
## 5｜贯穿推演与疑问闭合
```

禁止插入额外一级标题。

---

### §0｜概览与完整链路

**元信息表**：

| 顶层问句 | 推理模式 | 边界 |
|----------|----------|------|

**完整链路叙事**（必做，150–250 字）：用**一个连贯段落**从用户动作写到原理层，点名关键术语，说明「先…再…因此…」。不是 bullet 堆砌，是**可读的故事线**。

**有序主链表**（调用/时间顺序）：

| 步骤 | 环节 | 输入 | 输出 | 一句话 |
|------|------|------|------|--------|
| 1 | … | … | … | … |

**待回答疑问清单**（10–20 条，来自工作流 step 3）：

| # | 疑问 | 计划在哪层回答 |
|---|------|----------------|

---

### §1｜术语地基

每个关键术语一块。**固定六行**（比旧版多「何时出现」「如何实现」）：

```markdown
### {术语}

| | 内容 |
|--|------|
| **是什么** | … |
| **不是什么** | … |
| **为什么需要** | 没有它，链路在哪一步会断 |
| **在链路哪一步出现** | 对应 §0 主链第几步 |
| **如何做到** | 机制/数据结构/注册方式，≤80 字 |
| **依据** | 出处 |
```

技术栈主题 §1 不少于 **5** 个术语，覆盖问句中每个大词及读者必问概念（如：分派、分派键、计算图、Stream…）。

---

### §2｜路径对比

≥2 条正交路径时必做。表后说明**主链选哪条、逐步推演用哪条**。

| 路径 | 何时走 | 为什么选/不选 | 核心做法 | 依据 |

单路径写：「无并列选型，§2 从略。」

---

### §3｜逐层深入（核心）

**按 L0 → L4 顺序**；每层一个 `## L{n} · {层主题}`。

#### 每层固定结构（四层骨架，不可省略）

```markdown
## L{n} · {层主题}

**本层在整条链中的位置**  
{上一层的输出} → **【本层各步骤】** → {下一层的输入}

**本层要解决的疑问**（从 §0 清单摘入；写完后逐条标 ✅）  
1. …？ → 见步骤 k  
2. …？

---

### 步骤 1 · {环节名}

**为什么需要**  
{没有这一步会怎样；解决什么矛盾}

**是什么**  
{精确定义，无歧义}

**如何做到**  
{按顺序写：谁调用谁、关键数据结构、注册/配置/算法；可 3–6 个短句或编号步骤}

**接上一步 / 交下一步**  
- 上一步留下：…  
- 本步产出：…  
- 下一步消费：…

**例子**  
{可核对的具体场景}

**依据**  
{出处；推不出处标「待核实」且不进主链结论}

---

### 步骤 2 · …
（同模板）

---

**本层串讲**  
{80–150 字：仅本层，把各步骤连成一条线，用「因此/于是/再」连接}

**引向 L{n+1}**  
{本层仍解释不了、必须下钻的点，1–3 条}
```

#### 节点写法纪律

- **禁止**并列「知识卡片」式罗列；**必须**按 §0 主链的**时间/调用顺序**写步骤
- **禁止**节点表格中的：难度、易混辨析、下游（用户反馈无信息量）
- 关系/区别写进「是什么」「为什么需要」「如何做到」，不单开空泛字段
- 同一概念若在 L2 出现，L3 写**机制细节**，不重复 L2 定义（可写「定义见 L2 步骤 k」）
- 技术主题：L3 步骤须写到**可复述的实现级**（如：分派表、DispatchKeySet 合并规则、Stream 第四 launch 参数）
- 每层步骤数 **2–5**；全篇主链步骤 **15–25**

#### 疑问覆盖规则（最重要）

动笔前从 §0 清单取疑问；每层「本层要解决的疑问」必须**显式列出并逐条闭合**。

常见必答疑问类型（技术栈类按需选用）：

| 类型 | 必须回答 |
|------|----------|
| 存在性 | 为什么需要 X？没有 X 会怎样？ |
| 定义 | X 是什么？不是什么？ |
| 实现 | X 如何实现？关键数据结构/接口？ |
| 关系 | X 与 Y 什么关系？谁包含谁、谁调用谁？ |
| 时机 | 什么情况下需要 X？在链路哪一步？ |
| 流程 | X 如何构建/注册/发射/调度？ |
| 二次机制 | 为什么要再来一次（如二次分派）？ |

**禁止**留「详见某某」却不展开；**禁止**用「一般来说」代替机制。

---

### §4｜支链

未入主链的分支，每条含：**何时需要、与主链差在哪、为何主链先不讲**。

---

### §5｜贯穿推演与疑问闭合

**5.1 完整推演**（必做，300–500 字）：选一个具体例子，**按 §0 主链逐步编号**，每步写清：发生了什么、调用了什么、数据结构如何变、为何进入下一步。相当于带读者**走一遍调试器**。

**5.2 疑问闭合表**（必做）：§0 清单**每一条**对应「已答位置」或本段补答：

| # | 疑问 | 已答位置 | 一句话答案 |
|---|------|----------|------------|

未闭合项必须在 5.2 **补答**，不得留空。

**5.3 最易误解**（2–3 条）：误解 → 正确理解 → 依据步骤号。

---

## 依据协议

官方文档 > 规范 > 白皮书 > 经典教材/论文。关键机制 **≥2 权威来源** 交叉后写入主链。

## 风格

- 中文；**细节不丢，但按链路顺序组织**，避免同一事实在多处重复
- **为什么 + 如何** 优先于名词堆砌
- 单步「如何做到」可 80–150 字；整层串讲 80–150 字
- 段落输入：§0 前引述提炼（≤2 句）
